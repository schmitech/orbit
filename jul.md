### #206 apply nested config edits on adapter reload
**Author:** schmitech (remsy.schmilinsky@schmitech.ca)
**Date:** 2026-06-07 12:02:28
**Hash:** cf781d1e

**Description:**
#206 apply nested config edits on adapter reload

Editing a nested import (e.g. adapters/multimodal.yaml) and reloading via the admin UI had no effect until a full server restart.

The config loader's _import_cache keys each parsed import by its own mtime, so a parent import (adapters.yaml) returned its cached, fully-merged subtree — masking changes to child files whose parent mtime was unchanged. clear_config_cache() and reload_config() never flushed this cache.

Fix: flush _import_cache in reload_config() and clear_config_cache() so a reload always re-reads and re-merges every imported file from disk.

### Improve adapter reload code quality
**Author:** schmitech (remsy.schmilinsky@schmitech.ca)
**Date:** 2026-06-07 14:17:47
**Hash:** 58c78220

**Description:**
Improve adapter reload code quality

Addresses five bugs found during code review of the adapter lifecycle system:

- Initialization storm: _wait_for_adapter_initialization claimed the init lock then re-entered get_adapter while still holding it, causing a 60s timeout loop under concurrent load. Fix inlines the load/cache/release sequence directly in the fallback path instead of recursing.

- Stale datasource on reload: reload_adapter_configs() updated dependency_cleaner.config before cleanup ran, so invalidate_datasource built its cache key from the new datasource URL and missed the still-pooled old connection.

Fix defers update_config() to a finally block after the reloader completes.

- Incomplete FaultTolerantAdapterManager facade: ~10 public methods from DynamicAdapterManager were not delegated, causing AttributeError at runtime. Added explicit delegation for all missing methods and standardised the method name reload_adapters → reload_adapter_configs throughout.

- config_manager.py: nested imports resolved relative paths against the parent file's directory instead of the imported file's own directory; path-traversal guard used startswith('..') instead of commonpath.

- dependency_cache_cleaner.py: private_generate_cache_key/_datasource_pool access replaced with a new public DatasourceRegistry.invalidate_datasource() method; discarded template_collection_name call and redundant old_config fallback removed.

Also removes dead code (_is_enabled, unreachable sequential-fallback guard) and  returns copies from backward-compat cache properties to prevent external mutation.

  Adds 6 regression tests in test_reload_components.py covering the datasource
  config-isolation bug (before/after) and the reload_adapter_configs ordering
  contract.

### #212 continue button not showing
**Author:** schmitech (remsy.schmilinsky@schmitech.ca)
**Date:** 2026-06-11 12:50:37
**Hash:** 6230e3fe

**Description:**
#212 continue button not showing

The "continue"/thread button is gated on the backend including threading.supports_threading in the final SSE event. That payload is suppressed by _has_meaningful_results, whose first gate scanned the LLM's answer text for "no results" phrasing via _response_indicates_no_results.

The specific defect: Heuristic 2 in that function ran _no_results_regex.search() against the entire response with no length gate (response_processor.py:588). The regex matches phrases like not available, does not include, no records, not found, try another anywhere in the text. So a valid, data-rich customer-orders answer that happened to say something like "Tracking is not available yet" got flagged as "no results" → threading suppressed → no button. Since LLM wording varies between generations on the same data, the button appeared/disappeared intermittently.
